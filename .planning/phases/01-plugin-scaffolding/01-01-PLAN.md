The four allow rules are exactly as specified in D-04:

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Repo → Public | `plugin.json` and `settings.local.json.example` are committed to a public repo |
| settings.local.json | User's local permissions file — must not be committed |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | `settings.local.json.example` | mitigate | File contains zero secrets — only permission rule strings. No API keys, tokens, or credentials. Template-only content is safe to commit publicly. |
| T-01-02 | Information Disclosure | `settings.local.json` (user copy) | mitigate | `.gitignore` excludes `settings.local.json`. User's actual permissions file is never committed. Documented in README. |
| T-01-03 | Tampering | `plugin.json` | accept | Static config file. No executable surface. Malicious modification requires repo write access — out of threat model for a local developer plugin. |
| T-01-04 | Spoofing | `plugin.json` `name` field | accept | Plugin name `claude-skills` is used for install routing only. No authentication surface in Phase 1 — all skills are stubs. Naming collision with a malicious plugin requires the user to `plugin install` a different path. |
</threat_model>
